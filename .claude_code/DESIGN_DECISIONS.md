# 关键设计决策记录

> 每个决策记录：背景 → 选项 → 选择 → 理由 → 代价

---

## D1: 双 MCU 职责划分方案

**背景**: 控制器有两块 STM32G474，各带 2 CAN FD + 4 RS485。如何划分职责？

**选项**:
1. 方案一：运动控制域 vs 辅助设备域
2. 方案二：空间区域划分（左/右）
3. 方案三：主控制 vs 监控
4. 方案四：子系统划分（底盘 vs 机械臂/上装）

**选择**: **方案四（子系统划分）**

**理由**:
- 多关节机器人场景下，方案一将全部运动设备压在 MCU-A 2路CAN上，经典CAN场景过载 108%+
- 方案四使 4路CAN均衡参与运动控制，经典CAN下所有通道 <55%
- 双MCU硬件投入在方案四下获得对等回报
- 代码复用度从 ~60% 提升到 ~85%
- 底盘和机械臂可独立开发、测试、升级、换型

**代价**:
- BMS/IMU/传感器等需明确分配到子系统（已分配：BMS+IMU→MCU-A）
- 安全从集中式变为分布式，增加跨MCU互锁逻辑（~500 LOC）
- RK侧新增运动学协调器（~2000 LOC）
- MCU间GPIO从5根增加到8根

**触发切换的定量分析**: 经典CAN 1Mbps × 2路，承载7关节(各1ms周期帧) + 底盘(行走×2+转向×2+舵轮×2，各1ms帧) → 总负载 132%，超出 50% 安全线

## D2: SPI 拓扑

**背景**: RK3588 与两块 STM32 的 SPI 连接方式。

**选项**:
1. 两组独立 SPI（各占一组 SPI 控制器 + CS + DRDY）
2. 共享 SPI 总线（同 MOSI/MISO/SCLK，独立 CS）
3. 级联（RK 只接一块 MCU，另一块通过 MCU 间转发）

**选择**: **两组独立 SPI**

**理由**: 故障隔离是硬约束。共享 SPI 总线下一块 MCU 的 SPI 引脚短路拉低总线会导致另一块 MCU 同时失联。

**代价**: 多占用 RK3588 一组 SPI 控制器 + 4 个引脚

## D3: 同类型通道共用任务 vs 每通道独立任务

**背景**: 每块 STM32 需要管理 2 CAN + 4 RS485 通道，FreeRTOS 任务如何划分？

**选项**:
1. 每通道一个任务（2 CAN + 4 RS485 = 6通道任务）
2. 同类通道共用任务（1 CAN_RX + 1 CAN_TX + 1 RS485 = 3任务）

**选择**: **同类通道共用任务**

**理由**: STM32G474 RAM 有限（128KB），14个额外任务（模型一）的栈开销过大（~14KB）。CAN/RS485 接收由中断+DMA保证实时性，任务只处理已缓冲数据。

**代价**: 同任务内一个通道的处理延迟可能影响另一个通道的响应（通过限制单次处理帧数缓解：CAN每次最多8帧后 yield）

## D4: 协议序列化方案

**背景**: RK↔STM32 通信协议的序列化方式。

**选项**: 自定义定长二进制 / TLV / nanopb(Protobuf) / FlatBuffers / CBOR / MessagePack

**选择**: **自定义定长二进制（Header定长 + Payload变长）**

**理由**:
- 实时控制路径要求零解析开销（32字节Header直接memcpy/map）
- STM32G474 RAM紧张，nanopb/FlatBuffers编解码缓冲开销 2-4KB per codec
- 产品迭代中，版本兼容通过 flags 扩展字段满足，不需要通用序列化的灵活性

**代价**: 新增字段需手动管理兼容性（通过数据字典自动生成编解码代码缓解）

---

## D5: 外部看门狗策略

**背景**: 外部硬件 WDT 是监控一块还是两块 MCU？

**选项**:
1. 一个外部 WDT 同时监控两块 MCU（联合喂狗）
2. 两个独立外部 WDT 各自监控一块 MCU

**选择**: **V2 推荐选项二（两个独立 WDT）**

**理由**: 对齐"子系统独立"原则。底盘 WDT 复位不应导致机械臂也复位。机械臂 MCU 故障时，底盘可继续安全运行。

**代价**: 多一个外部 WDT 芯片，成本增加 ~$1

---

## D6: 跨子系统故障响应策略

**背景**: 一块 MCU 进入 FAULT 时，另一块 MCU 如何响应？

**选择**: **通知式而非指令式** — 故障 MCU 通过 GPIO 通知对方，接收方**自主评估**影响并决定响应级别。

**理由**: 
- 强制连锁停车违背"子系统独立"原则
- 例如：机械臂关节故障时，底盘完全可以降速继续行走（移动到安全位置）
- 接收方有更多上下文做出合理决策

**代价**: 接收方需要更复杂的故障评估逻辑

---

## D7: 硬件急停 vs 软件急停

**选择**: **严格区分**
- 硬件急停：物理切断全部驱动器使能电源（安全继电器），绕过所有 MCU 软件
- 软件急停：通过协议消息触发，仅触发子系统 EMERGENCY_STOP 状态
- 软件急停不能替代硬件急停

**理由**: 功能安全基本要求。软件可能在任何层级失效。

---

## D8: 代码角色差异管理

**背景**: MCU-A 和 MCU-B 代码如何管理差异？

**选择**: **配置结构体 + 接口注入 + 编译目标配置**，禁止 #ifdef MCU_A / MCU_B

**具体方式**:
```c
// 通过设备描述表注入差异
const DeviceDescriptor chassis_devices[] = { ... };  // mcu_a/config/
const DeviceDescriptor arm_devices[] = { ... };      // mcu_b/config/

// 公共代码使用统一接口
int device_manager_init(DeviceManager *mgr, 
                        const DeviceDescriptor *devices, 
                        int count);
```

**理由**: 条件编译维护两套业务不可扩展，增加第三个子系统时需要重构全部 #ifdef。
