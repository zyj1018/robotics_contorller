## Session 2: V1 方案评审 → 发现 CAN 总线瓶颈

**用户反馈**:
> "当前推荐的运动控制域+辅助设备域的分配方法。该方法缺点为：如果需要多个关节高频率的控制的情况下，两路CAN可能不够用（因为可能电机不支持CANFD只能用CAN）。我们设备两块MCU的目的就是为了可以有足够的CAN接口可以使用。所以我们需要使用方案：子系统划分 再设计一版"

**核心问题分析**:
- 方案一下全部运动设备集中在 MCU-A 的 2 路 CAN
- 当关节多（6-7轴）+ 驱动器仅支持经典 CAN 1Mbps → MCU-A CAN 过载 108%~132%
- MCU-B 的 CAN 资源闲置（负载 <5%）
- 双 MCU 硬件投入未在运动控制带宽上获得回报

**定量对比**:
| 场景 | V1 MCU-A CAN负载 | V2 最重CAN负载 |
|------|:--:|:--:|
| 经典CAN, 7关节+底盘 | **132%** ❌ | 54% ✅ |
| CAN FD, 7关节+底盘 | 32% | 23% |

## Session 3: V2 架构设计 — 子系统划分方案

**输入**: 用户明确要求使用方案四（子系统划分）

**完成内容**:
- 创建 `ARCHITECTURE_V2.md` (1253行，39章节)
- V2 采用 **方案四（底盘域 vs 机械臂/上装域）**

**V2 核心变化**:

| 维度 | V1 | V2 |
|------|----|----|
| MCU-A 角色 | 运动控制域 | **底盘子系统**（行走/转向/舵轮/BMS/IMU/超声） |
| MCU-B 角色 | 辅助设备域 | **机械臂/上装子系统**（关节1-7/夹爪/云台/升降） |
| CAN 利用 | MCU-A独占运动CAN | **4路 CAN 均衡参与运动控制** |
| MCU 关系 | 不对等 | **对等**（两MCU都是运动控制） |
| 代码复用 | ~60% | **~85%** |
| 安全 | 集中在MCU-A | **分布式**：子系统独立安全 + 跨MCU互锁 |
| 故障隔离 | MCU-B故障不影响MCU-A | MCU-B故障→MCU-A自主评估，底盘可继续降速 |

**V2 新增模块**:
- `rk3588/kinematics/` — 运动学协调器（底盘+臂轨迹协调）
- `mcu_common/control/` — 公共运动控制库（轨迹/限幅/PID，两MCU复用）
- 跨子系统安全互锁 GPIO（8根：心跳×2 + 安全×2 + 使能×2 + 子系统急停×2）
- 独立外部 WDT（每 MCU 一个，不复位对方）

**CAN 负载对比（关键结论）**:
| 通道 | 负载(经典CAN) | 负载(CAN FD) |
|------|:----------:|:--------:|
| CAN-A1 (底盘主驱动) | 54% | 16% |
| CAN-A2 (底盘扩展+BMS) | <5% | <2% |
| CAN-B1 (臂主驱动) | 48% | 14% |
| CAN-B2 (臂末端+上装) | 32% | 9% |
